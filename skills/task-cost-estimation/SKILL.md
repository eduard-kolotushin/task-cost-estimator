# Оценка трудозатрат (TaskTracker wiki)

## Когда опираться на навык

Запрос на **оценку задачи**, **декомпозицию с оценками**, **таблицу оценок** в wiki или сценарий «прочитать wiki → оценить → создать новую страницу».

## Инструменты

1. **get_wiki_page(code)** — загрузить исходную страницу; использовать `_extracted_wiki_page_body` и текст требований.
2. **get_wiki_hierarchy(spaces?)** — дерево wiki в пространстве(ах); в API всегда `root=null` (полный обзор). Для навигации по структуре страниц.
3. **create_wiki_page_estimation** — создать **новую** страницу с таблицей; передать `summary`, `space`, массив `rows`.
4. **update_wiki_page** — только если нужно поправить тело существующей страницы (редко).

Связь родитель–потомок **не** вызывать инструментом: при `single-run --parent-page CODE` связь выполняет среда после создания страницы.

## Таблица: 5 колонок

| Колонка | Правило |
|--------|---------|
| **Команда** | Роль / команда по строке (например Аналитика, RND). |
| **Компонент** | **Всегда** `VIEW` в каждой строке данных. |
| **Этап** | Фаза или поток работ. |
| **Оценка** | Только **чел.-дни**. **Сумма** оценок подзадач из **Комментарий** в этой строке. |
| **Комментарий** | Подзадачи с оценкой каждой в **чел.-дни**; **обоснование**, зачем нужен **именно такой** набор задач. |

Сумма подзадач (чел.-дни) в **Комментарий** должна совпадать с **Оценка**.

## Порядок действий

1. Вызвать **get_wiki_page** по коду исходной задачи.
2. Разложить работу на строки таблицы (по командам/этапам).
3. Вызвать **create_wiki_page_estimation** с новым заголовком `summary` и корректным `rows`.
4. Сообщить пользователю код созданной страницы из ответа API.

## Не делать

- Не ставить в **Компонент** ничего кроме `VIEW`.
- Не использовать часы вместо чел.-дней.
- Не заполнять **Комментарий** без явной связи подзадач и обоснования набора.
